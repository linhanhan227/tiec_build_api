# TieApi 项目修改记录

本文档记录了 TieApi 项目的主要修改和改进。

## 主要修改

### 1. 嵌入资源到二进制文件 (2026-01-27)

#### 背景
原本，`tiecc` 目录（包含编译器二进制文件）和 `stdlib` 目录（包含标准库文件）在构建时被复制到构建输出目录，运行时从那里复制到工作目录。这导致二进制文件依赖外部文件，不够自包含。

#### 修改内容
- **构建脚本 (`build.rs`)**：
  - 添加了 `zip` crate 作为构建依赖。
  - 修改为创建 `tiecc.zip` 和 `stdlib.zip`，包含相应目录的内容。
  - 使用 `cargo:rerun-if-changed` 确保目录变化时重新构建。

- **主程序 (`main.rs`)**：
  - 嵌入 ZIP 数据：`const TIECC_ZIP: &[u8]` 和 `const STDLIB_ZIP: &[u8]`。
  - 设置路径，但不在启动时解压（延迟到需要时）。

- **状态管理 (`state.rs`)**：
  - 在 `AppState` 中添加 `tiecc_dir`、`stdlib_dir`、`tiecc_zip`、`stdlib_zip` 字段。
  - 添加 `ensure_tiecc_extracted()` 和 `ensure_stdlib_extracted()` 方法，在需要时解压。

- **上传处理 (`api/upload.rs`)**：
  - 修改 stdlib 复制源从硬编码路径改为动态路径。
  - 在复制前调用 `ensure_stdlib_extracted()`。

- **工作器 (`worker/mod.rs`)**：
  - 更新 `copy_compiler_binary` 函数，接受 `AppState` 参数。
  - 在复制前调用 `ensure_tiecc_extracted()`。

- **工具函数 (`utils.rs`)**：
  - 添加 `extract_zip` 函数，用于解压 ZIP 并设置权限。

- **`.gitignore`**：
  - 添加 `src/*.zip` 以忽略生成的 ZIP 文件。

#### 优势
- **自包含**：二进制文件包含所有必要资源，无需外部依赖。
- **延迟解压**：仅在第一次使用时解压，提高启动效率。
- **按需加载**：只有当需要 tiecc 或 stdlib 时才解压，避免不必要的 I/O。
- **跨平台兼容**：ZIP 包含所有平台的二进制文件，运行时选择合适版本。

#### 技术细节
- 使用 `zip` crate 创建和解压 ZIP 文件。
- 解压时设置可执行权限（Unix 系统）。
- 路径处理使用 `std::path::Path` 确保跨平台兼容。

### 3. 添加接口速率限制 (2026-01-28)

#### 背景
为了防止 API 接口过载，需要添加速率限制机制，限制每个客户端的请求频率。

#### 修改内容
- **依赖 (`Cargo.toml`)**：
  - 添加 `actix-governor = "0.5"` 依赖，用于速率限制。

- **主程序 (`main.rs`)**：
  - 导入 `actix_governor::{Governor, GovernorConfigBuilder}` 和 `middleware::Logger`。
  - 配置 governor：每秒 10 个请求，突发最多 20 个。
  - 在 `App::new()` 中添加 `.wrap(Governor::new(&governor_conf))` 中间件。

#### 优势
- **防止过载**：限制请求频率，避免服务器资源耗尽。
- **公平性**：确保所有客户端都能公平访问。
- **可配置**：可以根据需要调整速率和突发限制。

#### 技术细节
- 使用 `actix-governor` crate 实现基于令牌桶算法的速率限制。
- 速率限制应用到所有路由，包括 API 和 Swagger UI。
- 当超过限制时，返回 429 Too Many Requests 状态码。

### 4. 任务队列溢出处理优化 (2026-01-28)

#### 背景
为了更好地处理高并发场景下的队列过载，需要优化任务队列的溢出处理策略，从取消前5个任务改为取消前60个任务，以更有效地释放队列空间。

#### 修改内容
- **状态管理 (`state.rs`)**：
  - 修改 `enqueue_task` 方法中的循环，从 `for _ in 0..5` 改为 `for _ in 0..60`。
  - 当队列达到 `QUEUE_CAPACITY`（默认100）时，新任务会触发前60个排队任务的取消逻辑。

#### 优势
- **更有效的过载保护**：一次性取消更多积压任务，为新请求让出更多空间。
- **减少队列阻塞**：在高负载情况下更快地恢复系统响应性。
- **可配置性**：取消数量可以通过修改代码中的数字轻松调整。

#### 技术细节
- 取消的任务状态会被设置为 `Cancelled`，并在数据库中更新。
- 取消操作是异步执行的，不会阻塞新任务的入队。
- 记录警告日志，便于监控和调试。

### 5. IP-based 速率限制和封禁机制 (2026-01-28)

#### 背景
为了防止恶意攻击和过载，需要实现基于IP的严格速率限制，当单个IP在一秒内访问超过120次时，将其封禁7天。

#### 修改内容
- **依赖 (`Cargo.toml`)**：
  - 添加 `rand = "0.8"` 依赖，用于随机清理操作。

- **中间件模块 (`src/middleware.rs`)**：
  - 新建 `IpRateLimiter` 结构体，实现基于IP的请求频率跟踪和封禁。
  - 实现 `IpRateLimiterMiddleware`，在每个请求中检查IP状态和频率限制。
  - 当IP在一秒内超过120次请求时，自动将其加入7天封禁列表。

- **主程序 (`main.rs`)**：
  - 导入新的中间件模块。
  - 将 `IpRateLimiter` 添加为应用中间件，置于 `Governor` 之前。
  - 修改Governor配置为每秒120个请求（与IP限制器保持一致）。

#### 优势
- **严格的安全控制**：防止单个IP进行DDoS攻击或滥用。
- **自动封禁**：违规IP会被自动封禁7天，无需人工干预。
- **内存高效**：定期清理过期的数据，避免内存泄漏。
- **双重保护**：结合actix-governor提供多层防护。

#### 技术细节
- 使用 `RwLock` 保护共享状态，支持并发访问。
- 每个IP维护独立的请求计数器，每秒重置一次。
- 封禁列表定期清理，移除过期的封禁记录。
- 返回适当的HTTP状态码：403（已封禁）、429（速率超限）。
- 记录详细的警告日志，便于监控和调试。

### 6. IP封禁持久化存储 (2026-01-28)

#### 背景
之前的IP封禁只存储在内存中，服务器重启后封禁记录会丢失。为了实现持久化封禁，需要将封禁IP存储在本地文件中。

#### 修改内容
- **中间件模块 (`src/middleware.rs`)**：
  - 修改 `IpRateLimiter` 结构体，添加 `ban_file_path` 字段。
  - 将 `new` 方法改为异步，在启动时加载现有的封禁IP文件。
  - 添加 `load_banned_ips` 方法，从 `ip_ban.txt` 文件加载封禁IP列表。
  - 添加 `save_banned_ips` 方法，将当前封禁IP保存到文件。
  - 修改 `check_and_update_rate` 方法，在封禁新IP时立即保存到文件。
  - 修改 `cleanup` 方法，在清理过期封禁后更新文件。

- **文件格式**：
  - 自动创建 `ip_ban.txt` 文件，一行一个IP地址。
  - 包含文件头注释，说明格式和生成时间。
  - 只保存仍在封禁期的IP地址。

#### 优势
- **持久化存储**：服务器重启后封禁记录仍然有效。
- **可人工管理**：管理员可以手动查看、编辑或删除封禁IP。
- **备份友好**：文件可以轻松备份和恢复。
- **透明度高**：封禁列表以明文格式存储，便于审查。

#### 技术细节
- 文件路径固定为 `ip_ban.txt`，位于服务器运行目录。
- 启动时自动加载现有封禁IP，保持连续性。
- 每次封禁新IP或清理过期封禁时，立即更新文件。
- 文件格式包含元数据注释，便于理解和管理。
- 支持注释行（以#开头）和空行，增加可读性。

### 7. 资源管理进一步优化 (2026-01-30)

#### 背景
之前的资源管理方案虽然实现了自包含，但仍需在运行时解压ZIP文件。为了简化部署和提高启动速度，改为在编译时生成内嵌资源清单，运行时直接从二进制内逐文件复制到本地目录。

#### 修改内容
- **构建脚本 (`build.rs`)**：
  - 移除ZIP打包逻辑，改为生成Rust代码清单。
  - 创建 `assets_manifest.rs` 文件，包含所有资源文件的内嵌数据。
  - 使用 `include_bytes!` 宏直接嵌入文件内容。
  - 保留文件权限信息，用于Unix系统。

- **依赖管理 (`Cargo.toml`)**：
  - 移除 `zip = "2.2"` 构建依赖。
  - 保留 `walkdir = "2.5"` 用于遍历源目录。

- **资源模块 (`src/assets.rs`)**：
  - 新建模块，定义 `EmbeddedAsset` 结构体。
  - 包含 `include!(concat!(env!("OUT_DIR"), "/assets_manifest.rs"))` 以导入生成的清单。

- **状态管理 (`state.rs`)**：
  - 移除ZIP相关字段和解压逻辑。
  - 修改 `ensure_assets_extracted` 方法，改为逐文件写入。
  - 使用 `ASSETS` 静态数组遍历所有内嵌资源。
  - 设置正确的文件权限（Unix系统）。

- **主程序 (`main.rs`)**：
  - 移除ZIP常量和导入。
  - 添加 `mod assets;` 模块导入。
  - 修改 `.tiec` 目录路径为可执行文件同目录。

- **工具函数 (`utils.rs`)**：
  - 移除未使用的 `extract_zip` 函数和相关导入。

#### 优势
- **更快的启动速度**：无需解压ZIP，直接文件I/O。
- **更简单的部署**：单二进制文件，无需外部资源。
- **更好的跨平台兼容**：直接处理文件权限，无需ZIP元数据。
- **减少依赖**：移除ZIP库，减小构建时间和二进制大小。

#### 技术细节
- 使用 `include_bytes!` 宏在编译时嵌入所有资源文件。
- 生成的清单包含路径、数据和权限信息。
- 运行时按需创建目录结构并写入文件。
- Unix系统上恢复原始文件权限。
- 路径处理使用 `std::path::Path` 确保跨平台兼容。

## 构建和运行

```bash
# 构建项目
cargo build

# 运行项目
cargo run
```

第一次运行时，会自动创建 `./.tiec` 目录并解压嵌入资源。

## 文件结构

```
.tiec/              # 运行时创建的目录
├── tiecc/          # 解压的编译器二进制文件
├── stdlib/         # 解压的标准库文件
├── uploads/        # 上传文件目录
└── tasks.db        # SQLite 数据库
```

## 依赖

- `zip = "2.2"` (构建依赖，用于创建 ZIP 文件)
- 其他依赖见 `Cargo.toml`

## 注意事项

- 确保运行目录有写权限，以创建 `.tiec` 目录。
- 如果需要更新嵌入资源，修改 `src/tiecc` 或 `src/stdlib` 后重新构建。
- 嵌入的 ZIP 文件在构建时生成，位于 `target/debug/build/.../out/`。

### 10. 上传解压与构建目录策略调整 (2026-01-30)

#### 修改内容
- **上传接口 (`src/api/upload.rs`)**：
  - `.tsp` 文件保存后立即解压到 `{upload_dir}/{file_id}_extracted`。
  - 兼容常见 MIME 类型与大小写扩展名校验。

- **构建接口 (`src/api/build.rs`)**：
  - 构建优先使用上传时生成的解压目录作为编译目录。
  - 若解压目录不存在，回退到 `.tsp` 文件路径。

- **工作器 (`src/worker/mod.rs`)**：
  - 若构建路径已是目录则跳过解压，直接作为项目根目录。
  - 不再创建 `tie_build_xxxxx` 临时目录。

#### 优势
- 构建与上传目录一致，避免重复解压。
- 路径更清晰，便于定位构建产物。

### 11. 安卓基本库复制逻辑调整 (2026-01-30)
